### 3.2 — Padrão de criação de novas features

**Hierarquia de extensão segura (do menos invasivo para o mais invasivo):**

1. **Plugin nativo Chamilo** → `public/plugin/[NomePlugin]/`
   - Usar `HookPluginInterface` para integrar com o ciclo de vida do LMS
   - Arquivo obrigatório: `plugin.php` com metadados do plugin
   - Documentar em CUSTOMIZATIONS.md antes de criar

2. **Bundle Symfony próprio** → `src/[NomeProjeto]Bundle/`
   - Para features que precisam de Entities, Controllers, Services
   - Registrar em `config/bundles.php`
   - Entities separadas das do CoreBundle

3. **Vue component customizado** → `assets/vue/components/custom/`
   - Importar em componentes existentes via composition API
   - Não modificar componentes do core diretamente

4. **Override de configuração** → `config/settings_overrides.yaml`
   - Para parâmetros Symfony que precisam ser sobrescritos
   - Nunca editar `config/packages/` (chmod 555)

5. **Tema visual** → `assets/css/themes/[nome-tema]/`
   - Sobrescrever variáveis CSS do Tailwind
   - Não modificar `tailwind.config.js` diretamente

### 3.3 — Fluxo obrigatório para novas Entities (banco de dados)

```bash
# 1. Criar/modificar a Entity em src/[Bundle]/Entity/
# 2. Gerar migration (NUNCA deixar o Doctrine alterar schema diretamente)
php bin/console doctrine:migrations:diff

# 3. Revisar a migration gerada em src/Migrations/
# Verificar se há DROP TABLE ou ALTER TABLE que possam perder dados

# 4. Backup antes de executar
mysqldump -u chamilo -pchamilo_pass \
  --socket=/home/runner/mysql_run/mysql.sock chamilo \
  > /tmp/backup_$(date +%Y%m%d_%H%M%S).sql

# 5. Executar migration
php bin/console doctrine:migrations:migrate --no-interaction

# 6. Verificar resultado
php bin/console doctrine:migrations:status
```
